# Полный ручной пайплайн sandbox

Документ нужен как пошаговая шпаргалка для русского интерфейса COMPASS AI. Ничего не запускается из терминала: все делается руками в браузере.

Главная цель: создать кастомный dev-датасет, проверить его, обучить все модели, проверить ONNX-экспорт, протестировать назначение задач с AI-объяснениями и сформировать отчеты.

## 0. Важные правила перед стартом

1. Работай только через интерфейс.
2. Импорт данных пропускаем.
3. Везде, где есть выбор датасета, выбирай датасет из этого документа: `dev_custom_huge_20260710`.
4. Для первого полного прохода используй режим `Очень большой` и значения из раздела генерации ниже.
5. Абсолютные backend-лимиты намного выше, но их не надо ставить для первого полного end-to-end запуска: можно получить гигантские файлы и очень долгое обучение.
6. В кастомной схеме итоговый список полей должен быть ровно как в этом документе. Если в интерфейсе уже есть поле с таким же именем, его не добавляй второй раз, а просто проверь тип, значения, min/max и галочку `обязательно`.
7. Если в кастомной схеме есть лишние поля, которых нет в таблицах ниже, удали их кнопкой `×`.
8. Кнопку `Сбросить к общим полям` не нажимай.

Открой вкладку `Генерация данных`.

### 1.1. Основные поля генерации

Заполни верхнюю форму так:

| Поле в интерфейсе | Что выбрать или вписать |
|---|---|
| `ID датасета` | `dev_custom_huge_20260710` |
| `Профиль предметной области` | `custom` |

| `Размер датасета` | `Очень большой` |
| `Количество сотрудников` | `100` |
| `Количество задач` | `10000` |
| `Количество проектов` | `5` |
| `История на сотрудника` | `60` |
| `Пар для обучения` | `1000000` |
| `Кандидатов на задачу` | `20` |
| `Цель оптимизации` | `balanced` |
| `заменить существующий датасет` | галочка включена |

После выбора размера `Очень большой` появится блок `Подтверждение большого запуска`.

В нем поставь галочку:

| Поле в интерфейсе | Что сделать |
|---|---|
| `я понимаю размер запуска` | галочка включена |

Почему именно эти значения: это текущий большой пресет интерфейса. Он уже достаточно большой для обучения всех моделей и проверки распределения, но не превращает первый запуск в стресс-тест на десятки миллионов строк.

Если потом отдельно захочешь именно стресс-тест, backend для режима `Очень большой` принимает до `100000` сотрудников, `10000000` задач, `50000` истории на сотрудника, `100000000` пар и `10000` кандидатов на задачу. Для текущего пайплайна так не делай.

2. тип;
3. `values CSV`, если тип `categorical`;
4. `min` и `max`, если тип `numeric`;

Если поле есть, но его нет в таблицах ниже, удали его кнопкой `×`.

Если поля из таблицы ниже нет, нажми `Добавить поле` в нужном блоке и заполни новую строку.

### 2.2. Что проверить по очереди

2. `Задачи`

Проверь, что:

1. есть нейтральные ID сотрудников, а не важные для модели имена;

4. в кастомных полях есть `business_impact`, `technical_risk`, `customer_visibility`, `requires_code_review`, `requires_on_call`, `delivery_area`, `architecture_scope`, `test_depth`, `compliance_tags`.

### 2.5. Что должно быть видно в `История выполнений`

### 2.6. Что должно быть видно в `Пары для обучения`

Проверь, что:

2. есть пары задача-кандидат;
3. есть признаки совпадения навыков, нагрузки, риска и результата;
4. эта таблица не пустая.

Дополнительно можно открыть `Сводка`, `Графики` и `Канбан`, чтобы быстро проверить распределение статусов, ролей, уровней и задач.

## 4. Вкладка `Обучение`

Открой вкладку `Обучение`.

### 4.1. Основные параметры обучения

Заполни так:

| Поле в интерфейсе | Что выбрать или вписать |
|---|---|
| `Датасет` | `dev_custom_huge_20260710` |
| `Цель модели` | `balanced` |
| `Seed` | `19001` |
| `Максимум пар` | `120000` |
| `Доля обучения` | `0.70` |
| `Доля проверки` | `0.15` |
| `Доля теста` | `0.15` |
| `перестроить признаки` | галочка включена |

Почему `Максимум пар` ставим `120000`: это безопасный режим для MacBook Air M2 с 8 ГБ памяти. Датасет все равно остается большим и репрезентативным, но обучение не пытается одновременно обработать все `1000000` пар и не выбивает систему по памяти.

Нажми `Выбрать все`, даже если визуально уже все галочки стоят.

2. `sgd_classifier`
3. `logistic_regression`
4. `random_forest`
5. `hist_gradient_boosting`
6. `torch_mlp`

Оставь значения по умолчанию:

| Поле | Значение |
|---|---|
| `logistic max_iter` | `500` |
| `rf n_estimators` | `80` |
| `rf max_depth` | пусто |
| `sgd alpha` | `0.0001` |
| `sgd max_iter` | `1000` |

| `torch hidden` | `64` |
| `torch dropout` | `0.10` |

Для первого полного запуска эти параметры лучше не менять: они дают нормальное сравнение всех методов без ручной подгонки.

## 5. Вкладка `Модели`

Открой вкладку `Модели`.

### 5.1. Выбор сессии и модели

В поле `Сессия обучения` выбери последнюю сессию, которая была создана на датасете:

```text
dev_custom_huge_20260710
```

В поле `Модель` по очереди выбирай каждую модель:

1. `baseline_rule_based`
2. `sgd_classifier`
3. `logistic_regression`
4. `random_forest`
5. `hist_gradient_boosting`
6. `torch_mlp`

Для каждой модели нажимай:

1. `Детали модели`;
2. `Проверить`.

### 5.2. ONNX

Для ONNX-проверки:

1. выбери сессию обучения;
2. выбери модель;
3. в поле `Размер проверки` оставь `100`;
4. включи галочку `попробовать экспорт в ONNX`;
5. нажми `Экспорт / проверка`;
6. дождись результата;
7. повтори для остальных моделей, где нужен экспорт.

Если какая-то модель не экспортируется в ONNX, это не обязательно означает, что обучение сломалось. Главное смотри статус проверки модели и наличие артефакта. ONNX зависит от поддержки конкретного метода и установленных зависимостей.

### 5.3. Как выбрать лучшую модель для назначения задач

Для назначения задач обычно выбирай модель с лучшим балансом метрик на проверке.

Практический порядок:

1. сначала смотри `hist_gradient_boosting`;
2. затем `random_forest`;
3. затем `logistic_regression`;
4. если `torch_mlp` дал лучше метрики и проверка прошла, можно выбрать его;
5. `baseline_rule_based` используй как контрольную простую модель, а не как основной вариант.

### 6.3. Загрузить задачи

Нажми `Загрузить задачи`.

2. score/оценку;
3. причины выбора;
4. риски;
5. объяснение через Qwen/Ollama или локальное объяснение.

### 6.5. Распределить все задачи

После проверки одной задачи нажми `Распределить все задачи`.

Дождись прогресс-уведомления.

Что посмотреть после завершения:

1. кому назначены задачи;
2. нет ли сильного перекоса нагрузки;
3. какие задачи получили предупреждения по риску;
4. есть ли объяснения;
5. график нагрузки;
6. fairness/справедливость распределения;
7. kanban или список результатов.

Если результат выглядит слишком рискованным, повтори распределение с режимом:

```text
Осторожнее к рискам
```

Если хочешь максимальное качество без сильного учета нагрузки, попробуй:

```text
Лучшее качество
```

## 8. Вкладка `Канбан-лаборатория`

Открой вкладку `Канбан-лаборатория`.

Эта вкладка нужна для экспериментов. Здесь можно взять копию проверочного набора или создать отдельную лабораторную копию из датасета, поменять статусы задач на канбан-доске, удалить лишние задачи, добавить свои ручные задачи и только после этого отправить текущее состояние доски в модель.

Важно: исходный датасет и исходный проверочный набор не меняются. Все перемещения карточек, очистка столбцов и ручные задачи живут в копии внутри лаборатории. Если нажать `Сбросить изменения`, доска вернется к состоянию, которое было загружено из выбранного набора.

### 8.1. Загрузить копию для эксперимента

В верхнем блоке `Источник лаборатории` заполни:

| Поле | Что выбрать |
|---|---|
| `Проверочный набор` | `dev_custom_huge_eval` или лабораторную копию, если она уже есть |
| `Датасет для новой копии` | `dev_custom_huge_20260710 · generated` |
| `Сессия обучения` | последняя успешная сессия обучения на `dev_custom_huge_20260710` |
| `Модель` | сначала выбери лучшую модель по метрикам; если сомневаешься, выбери `logistic_regression` |
| `Режим рекомендации` | `Сбалансированно` |
| `Кандидатов на задачу` | `3` |

Галочку `Добавлять LLM-объяснение при раскрытии задачи`:

| Ситуация | Что делать |
|---|---|
| Хочешь видеть человеческие объяснения по раскрытой задаче | поставь галочку |
| Хочешь быстрее двигать карточки и смотреть только кандидатов | галочку не ставь |

Если `dev_custom_huge_eval` уже есть, нажми:

```text
Загрузить копию набора
```

Если хочешь создать отдельную лабораторную копию именно из датасета, нажми:

```text
Создать лабораторную копию из датасета
```

Будет создан набор с суффиксом:

```text
_kanban_lab
```

Например:

```text
dev_custom_huge_20260710_kanban_lab
```

После загрузки проверь верхние счетчики:

1. `Сотрудники` должен быть больше `0`;
2. `Задачи на доске` должен быть больше `0`;
3. `К рекомендации` показывает задачи из рабочих статусов;
4. `Рекомендации` сначала будет `0`, пока ты не нажмешь расчет.

### 8.2. Сохраненные лабораторные доски

Сохраненные доски Канбан-лаборатории лежат отдельно от обычных датасетов:

```text
sandbox_app/data/kanban_lab/
```

Они не лежат в:

```text
sandbox_app/data/generated/
```

и поэтому не появляются как обычные датасеты в `Просмотр данных`, `Обучение` или `Назначение задач`.

Это сделано специально: сохраненная лабораторная доска нужна только для вкладки `Канбан-лаборатория`.

Чтобы сохранить текущий эксперимент:

1. в поле `Название лабораторной доски` впиши понятное имя;
2. например:

```text
dev custom huge kanban experiment 01
```

3. нажми:

```text
Сохранить текущую доску
```

Сохранится текущее состояние:

1. сотрудники;
2. задачи;
3. текущие статусы задач на канбане;
4. ручные задачи;
5. удаленные из копии задачи уже не попадут в сохранение;
6. настройки выбранной модели и режима.

Рекомендации и LLM-объяснения лучше считать заново после загрузки, поэтому основная ценность сохранения — именно измененная доска.

Чтобы продолжить работу с сохраненной доской:

1. в поле `Сохраненная доска` выбери нужную доску;
2. нажми:

```text
Загрузить сохраненную доску
```

После загрузки можно снова двигать карточки, добавлять задачи и нажимать:

```text
Принять канбан и рассчитать рекомендации
```

Чтобы удалить сохраненную лабораторную доску:

1. выбери ее в поле `Сохраненная доска`;
2. нажми:

```text
Удалить сохраненную
```

Удаляется только сохраненный лабораторный артефакт из `sandbox_app/data/kanban_lab/`. Исходные датасеты, test cases, модели и отчеты не затрагиваются.

### 8.3. Как двигать задачи на канбане

На доске есть столбцы:

```text
К выполнению
В работе
Проверка
Готово
Заблокировано
Не выполнено
```

Чтобы перенести одну задачу:

1. зажми карточку мышкой;
2. перетащи ее в нужный столбец;
3. отпусти карточку.

После любого переноса старые рекомендации на карточках сбрасываются. Это нормально: ты изменил входные данные, поэтому нужно заново нажать расчет.

Чтобы перенести целый столбец:

1. в верхней части нужного столбца, сразу под заголовком, выбери новый статус в выпадающем списке;
2. нажми `Перенести столбец`.

Пример: если хочешь считать все задачи из `Проверка` снова задачами к работе, в столбце `Проверка` выбери:

```text
К выполнению
```

и нажми:

```text
Перенести столбец
```

Чтобы очистить столбец:

1. нажми `Очистить` в верхней части нужного столбца;
2. подтверди удаление.

Это удаляет задачи только из лабораторной копии. В исходном датасете они остаются.

### 8.4. Добавить ручную задачу в To Do

В блоке `Добавить ручную задачу` заполни поля.

Важно: `Тип задачи`, `Проект` и `Теги / навыки` теперь не нужно писать вручную текстом. Их надо выбирать из существующих значений текущего загруженного набора. Это сделано специально, чтобы не добавить задачу с тегом или проектом, которых модель не видела.

Если в форме вместо значений написано `Сначала загрузите набор`, значит сначала нажми `Загрузить копию набора` или `Создать лабораторную копию из датасета`.

Для теста с разработчиками можно заполнить так:

| Поле | Значение |
|---|---|
| `Название` | `Manual backend security review` |
| `Тип задачи` | выбери из списка существующий тип, например `backend`, `security`, `qa`, `integration` или другой доступный в твоем наборе |
| `Приоритет` | `high` |
| `Сложность 0..1` | `0.65` |
| `Оценка часов` | `12` |
| `Проект` | выбери из списка существующий проект, например `project_001` или другой доступный в твоем наборе |
| `Теги / навыки` | выдели один или несколько существующих навыков из списка, например `python`, `fastapi`, `security`, если они есть в текущем наборе |

Потом нажми:

```text
Добавить в To Do
```

Новая задача появится в столбце `К выполнению`.

Правила для ручных задач:

1. `Теги / навыки` выбирай только из списка; руками через запятую их больше не вписывай;
2. `Сложность 0..1` ставь от `0.1` до `1.0`;
3. `Оценка часов` ставь реалистично, например `4`, `8`, `12`, `24`;
4. `Тип задачи` и `Проект` выбирай из списка, чтобы ручная задача оставалась совместимой с текущим датасетом и обученными моделями.

### 8.5. Изменить персонал лабораторной копии

Персонал в `Канбан-лаборатории` тоже меняется только в копии. Исходный датасет из `sandbox_app/data/generated/` не меняется.

В блоке `Персонал лаборатории` можно:

1. добавить нового сотрудника;
2. удалить одного выбранного сотрудника;
3. удалить всех сотрудников из лабораторной копии.

Чтобы добавить нового сотрудника:

1. в поле `ID / название нового сотрудника` впиши нейтральный ID или оставь поле пустым;
2. если поле оставить пустым, интерфейс сам создаст ID вида:

```text
emp_lab_0001
```

3. в поле `Роль` выбери существующую роль из текущего набора;
4. в поле `Грейд` выбери существующий грейд из текущего набора;
5. в `Навыки сотрудника` отметь чекбоксами один или несколько существующих навыков;
6. метрики можно оставить такими:

| Поле | Значение |
|---|---|
| `Доступность 0..1` | `0.85` |
| `Текущая нагрузка 0..1` | `0.25` |
| `Усталость 0..1` | `0.2` |
| `Качество 0..1` | `0.8` |
| `Скорость 0..1` | `0.75` |
| `Надежность дедлайнов 0..1` | `0.8` |

7. нажми:

```text
Добавить сотрудника
```

Чтобы удалить одного сотрудника:

1. найди нужного сотрудника в сетке карточек ниже формы;
2. в его карточке нажми:

```text
Удалить
```

Чтобы удалить всех сотрудников:

1. нажми:

```text
Удалить всех сотрудников
```

2. подтверди действие.

После любых изменений персонала старые рекомендации сбрасываются. Это нормально: состав команды изменился, значит модель должна пересчитать назначения заново.

### 8.6. Рассчитать рекомендации по текущей доске

Когда расставишь доску как надо, нажми:

```text
Принять канбан и рассчитать рекомендации
```

Модель увидит именно текущее состояние копии:

1. задачи, которые ты перенес;
2. задачи, которые ты удалил из копии;
3. ручные задачи, которые ты добавил;
4. сотрудников, которых ты добавил или удалил;
5. текущие статусы карточек.

Рекомендации считаются для рабочих статусов:

```text
todo, in_progress, review, blocked
```

Задачи в `Готово` и `Не выполнено` не отправляются на рекомендацию.

После расчета на карточках появятся мини-кандидаты:

```text
#1 emp_...
#2 emp_...
#3 emp_...
```

Если карточка без кандидатов, значит для нее рекомендации еще не считались или она находится в статусе, который не отправляется на расчет.

### 8.7. Посмотреть задачу подробно

Нажми на любую карточку.

Справа откроется блок с деталями:

1. требования задачи;
2. приоритет;
3. сложность;
4. оценка часов;
5. проект;
6. требуемые навыки;
7. кандидаты модели;
8. график соответствия задачи и кандидатов;
9. LLM-объяснение, если галочка включена.

Если хочешь быстро поменять статус выбранной задачи без перетаскивания, нажми:

```text
Следующий статус
```

Если задача в эксперименте больше не нужна, нажми:

```text
Удалить задачу
```

Это удаляет задачу только из лабораторной копии.

### 8.8. Проверить LLM-объяснения

Если включена галочка:

```text
Добавлять LLM-объяснение при раскрытии задачи
```

то при раскрытии карточки появится блок `LLM-объяснение`.

Проверь, что там есть:

1. общий вывод по задаче;
2. объяснение по каждому кандидату;
3. цифры `score`, навыков, качества, скорости, риска, нагрузки или доступности;
4. сильные стороны;
5. риски;
6. нормальный текст без сырых JSON-скобок.

### 8.9. Сбросить эксперимент

Если хочешь вернуться к состоянию, которое было загружено из набора, нажми:

```text
Сбросить изменения
```

После этого:

1. ручные задачи исчезнут;
2. переносы между столбцами откатятся;
3. удаленные в лаборатории задачи вернутся;
4. добавленные вручную сотрудники исчезнут;
5. удаленные в лаборатории сотрудники вернутся;
6. рекомендации очистятся;
7. исходный датасет все равно останется нетронутым.
